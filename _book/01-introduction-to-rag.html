1  1. Introduction to RAG

1.1 1.1 What Is Retrieval-Augmented Generation

Retrieval-Augmented Generation (RAG) is an application pattern in which a language model receives external information at inference time and uses that information to produce an answer. Rather than relying only on parameters learned during pretraining, the model is given a dynamically retrieved context window that is relevant to the current query.

At a high level, a RAG system has three stages: retrieve, augment, generate. A user question is transformed into one or more retrieval queries; relevant passages or records are fetched from a knowledge source; then the model is prompted to answer using those materials. The key benefit is that the knowledge source can be updated independently of the model itself.

1.2 1.2 Core Principles and System Architecture

A practical RAG system usually contains the following components:

  • a data pipeline that ingests documents, records, or events;
  • a transformation layer that cleans, chunks, and annotates content;
  • an indexing layer that stores vectors, tokens, or structured fields;
  • a retrieval layer that selects candidate evidence;
  • a ranking layer that orders or filters candidates;
  • a generation layer that produces the final answer;
  • an evaluation and observability layer that measures quality.

The architecture matters because errors compound across layers. If chunking is poor, retrieval quality suffers. If retrieval is weak, prompting cannot rescue answer faithfulness. If evaluation is absent, teams often optimize for style instead of correctness.

1.3 1.3 Evolution of RAG Techniques

Early RAG implementations were mostly “semantic search plus prompt stuffing.” They embedded chunks, retrieved the top k, and placed them into a template. That pattern still works for many small systems, but the field has evolved rapidly.

Modern systems often include hybrid retrieval, metadata filtering, multi-stage ranking, context compression, tool use, query routing, and domain-specific structured retrieval. The trend is clear: the retrieval stack is becoming more adaptive, and the generation stack is becoming more constrained and instrumented.

1.4 1.4 RAG vs. Fine-Tuning

RAG and fine-tuning solve different problems.

RAG is strongest when the challenge is access to external knowledge: product catalogs, internal documentation, support articles, contracts, policies, and rapidly changing facts. Fine-tuning is strongest when the challenge is behavior: tone, style, task formatting, response patterns, or domain-specific output conventions.

In practice, many mature systems combine the two. A model may be tuned to follow a task reliably while RAG provides fresh or private facts. The common mistake is assuming one replaces the other in all cases.

1.5 1.5 When to Use RAG: Use Cases and Risk Levels

RAG is especially useful in:

  • enterprise knowledge assistants;
  • customer support copilots;
  • legal and policy lookup tools;
  • research and synthesis systems;
  • analytics assistants backed by tables or warehouses;
  • multimodal search over documents, images, or media.

Risk level matters. In low-risk settings, approximate retrieval may be acceptable. In high-risk settings such as healthcare, legal review, or compliance, the system needs stricter grounding, stronger auditability, and explicit fallbacks when evidence is weak.

1.6 1.6 Getting Started with RAG Systems

The best way to begin is not with the fanciest architecture. Start with a narrow use case, a clear corpus, and a measurable task. Build a baseline pipeline:

  1. Ingest a bounded set of documents.
  2. Normalize and chunk them.
  3. Embed and index the chunks.
  4. Retrieve top results for a test query set.
  5. Prompt the model with retrieved context.
  6. Evaluate answer quality and failure modes.

This baseline reveals where the real bottleneck lives. For many teams, the first bottleneck is not model choice but document quality and query coverage.

1.7 1.7 Evaluation Challenges and Optimization Directions

RAG systems are difficult to evaluate because the output quality depends on both retrieval quality and generation quality. A weak answer may come from poor evidence, poor prompt design, or both. That means metrics need to separate stages whenever possible.

Optimization typically moves along these axes:

  • improve corpus coverage;
  • improve chunking and metadata;
  • improve embedding/index selection;
  • improve candidate ranking and filtering;
  • improve prompt structure and answer constraints;
  • reduce latency and token cost.

1.8 1.8 Is RAG Dead? Debates and Emerging Alternatives

Claims that “RAG is dead” usually reflect a narrower point: long-context models, tool use, and stronger base models reduce the need for naive retrieval pipelines. They do not eliminate the need for retrieval itself.

In reality, the question is shifting from “Should we use RAG?” to “What retrieval strategy matches this task?” Sometimes the answer is classic vector search. Sometimes it is SQL generation, graph traversal, API calls, or an orchestrated combination of them.

1.9 1.9 A Minimal RAG Implementation

The smallest useful RAG system can be expressed in pseudocode:

documents -> clean -> chunk -> embed -> index
user query -> embed -> retrieve top_k chunks
top_k chunks + instructions + question -> language model
language model -> answer with citations or grounded summary

Frameworks such as LangChain and LlamaIndex can accelerate assembly, while local model runners such as Ollama can help with experimentation. Still, the framework is not the system. Good results come from thoughtful data and evaluation choices, not only from wiring together abstractions.

1.9.1 A Local Demo with Ollama

To make the architecture concrete, this book uses a local-first setup based on Ollama and turns the book itself into the knowledge base:

  • generator model: qwen3.5:2b
  • embedding model: bge-m3:latest
  • ingestion script: code/ingest_book.py
  • indexing script: code/build_book_index.py
  • question-answering script: code/ask_book.py

The workflow is intentionally practical:

  1. read the .qmd chapters listed in _quarto.yml;
  2. chunk them into retrieval units with source-path metadata;
  3. embed those chunks with bge-m3:latest;
  4. store the result as a local JSON index;
  5. answer questions against the book with qwen3.5:2b.

1.9.2 Running the Example

Before running the script, pull the models locally:

ollama pull qwen3.5:2b
ollama pull bge-m3:latest

Then build and query the book index:

python3 code/ingest_book.py
python3 code/build_book_index.py
python3 code/ask_book.py --question "What does the book say about hybrid retrieval?"

1.9.3 Core Code Path

The key ingestion step turns book chapters into reusable retrieval chunks:

def build_corpus() -> list[dict[str, str]]:
    corpus: list[dict[str, str]] = []
    for document in load_book_documents():
        corpus.extend(chunk_document(document["path"], document["text"]))
    return corpus

The answering step then retrieves matching chunks from that locally built index and passes them to the generator. This keeps the example grounded in a real artifact: if the chapter text changes, the corpus and answers can change too.

1.9.4 Why This Example Matters

A project like this is useful for two reasons. First, it gives the reader a baseline that can run entirely on local models. Second, it answers a real question: how do we build a RAG system over a body of source material that we actually care about?

In a real project, the next upgrades would usually be better chunking, metadata-aware filtering, a persistent vector store, and stronger evaluation. Those improvements make more sense once a baseline exists and can already answer questions over the full book.
