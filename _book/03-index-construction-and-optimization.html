3  Index Construction and Optimization

3.1 From Chunks to Searchable Units

At the end of Chapter 2, we had a set of chunks that were clean enough and small enough to retrieve. But those chunks are still just text. The system still needs a way to search them well.

That is the job of indexing.

In plain language, indexing means turning chunks into a form that the retrieval step can search quickly and usefully. In this project, that mostly means creating embeddings for each chunk and saving them in an index file.

So the flow now looks like this:

raw chapters
-> cleaned and chunked text
-> embeddings for each chunk
-> searchable index
-> retrieval at question time

This chapter is about that middle step.

3.2 What This Chapter Is Trying to Solve

The main problem is simple: even if the corpus is well prepared, the system still needs a good way to find the right chunks for a new question.

That means this chapter is trying to answer questions like:

  • how do we represent chunk meaning in a searchable form;
  • how does the system compare a user question with stored chunks;
  • what is the difference between an embedding model, an index, and a vector database;
  • when is a simple local index enough, and when is a bigger system needed.

If Chapter 2 was about making good chunks, Chapter 3 is about making those chunks searchable.

3.3 Why Plain Text Search Is Not Enough

Suppose a chunk says:

RAG is strongest when the problem is access to external knowledge.

A user might ask:

When should I use RAG instead of fine-tuning?

The wording is different, but the meaning is related. Simple keyword search may do fine in some cases, but it can also miss useful matches when the query and the chunk use different words.

That is why dense retrieval exists. Instead of matching only on surface words, it tries to match on semantic similarity.

This does not mean keyword methods stop mattering. It means we often need another tool in addition to exact term matching.

3.4 Embeddings and Their Role in RAG

An embedding is a numeric representation of text. The practical idea is that chunks with similar meaning should end up closer together in that numeric space.

In a RAG system, embeddings are useful because they let us do this:

  1. turn each chunk into a vector;
  2. turn the user question into a vector;
  3. compare those vectors;
  4. rank chunks by similarity.

That is the core of dense retrieval.

The benefit is easy to understand: the system may still retrieve a relevant chunk even when the user does not use the exact same wording as the document.

The limit is also important: embeddings do not “understand” truth. They only encode patterns of similarity learned during training. If the embedding model is weak for the domain, retrieval quality will suffer.

3.5 What an Embedding Model Actually Helps With

It helps to be concrete here.

An embedding model is good at things like:

  • matching related wording;
  • handling paraphrases;
  • finding semantically similar passages;
  • supporting multilingual or cross-domain matching, if the model is trained for that.

An embedding model is less reliable for things like:

  • exact identifiers;
  • strict keyword matching;
  • small but important wording differences;
  • structured constraints such as dates, IDs, or access rules.

That is why embeddings are powerful, but not sufficient by themselves.

3.6 How This Project Builds an Index

In this repository, indexing is intentionally simple. After the book is chunked, the system sends each chunk to an embedding model and stores the result in a JSON file.

The key script is code/cli/build_book_index.py, and the core embedding call lives in code/core/book_rag.py:

def ollama_embed(texts: list[str], embed_model: str = EMBED_MODEL) -> list[list[float]]:
    model_name = normalize_cloud_model_name(embed_model) if is_cloud_model(embed_model) else embed_model
    response = ollama_post("/api/embed", {"model": model_name, "input": texts}, model_name=embed_model)
    return response["embeddings"]

That function takes text chunks, sends them to Ollama’s embedding endpoint, and gets back vectors.

The index-building script then stores each chunk together with:

  • its chunk id;
  • its source path;
  • its section;
  • its text;
  • its embedding.

So the index is not only a list of vectors. It is a list of vectors tied back to the original chunk content and metadata.

3.7 Why the Current Index Is Simple but Useful

The current design stores embeddings in a JSON file such as code/build/book_index_*.json.

That is not the most scalable design, but it is useful for learning because:

  • the whole process stays visible;
  • the stored data is easy to inspect;
  • there is little hidden infrastructure;
  • the connection between chunk text and embedding stays obvious.

This is a good baseline for a teaching project. You can see the data, understand the flow, and debug it without needing a separate vector database.

3.8 Dense Retrieval in Plain Language

Once the chunks are embedded, the system can compare a question embedding with each stored chunk embedding.

In this project, that comparison is done with cosine similarity:

def cosine_similarity(vec_a: list[float], vec_b: list[float]) -> float:
    numerator = sum(a * b for a, b in zip(vec_a, vec_b))
    norm_a = math.sqrt(sum(a * a for a in vec_a))
    norm_b = math.sqrt(sum(b * b for b in vec_b))
    if norm_a == 0 or norm_b == 0:
        return 0.0
    return numerator / (norm_a * norm_b)

You do not need to worry too much about the math here. The practical meaning is enough: the closer the vectors point in the same direction, the more similar the system considers the texts.

That gives the system a way to rank chunks by semantic closeness to the user’s question.

3.9 Why Embeddings Alone Are Not the Whole Story

It is easy to overtrust embeddings once they start working.

But dense retrieval still has common failure cases:

  • it may miss exact keywords that matter;
  • it may rank vaguely related text too highly;
  • it may struggle with codes, IDs, or product names;
  • it may not respect business rules such as time, tenant, or permissions.

That is one reason hybrid retrieval is useful. Dense retrieval is good at semantic matching. Sparse retrieval is often better at exact term matching. Using both can produce more stable results.

So the right mental model is:

  • embeddings help the system search by meaning;
  • they do not remove the need for other retrieval signals.

3.10 How to Choose an Embedding Model

Model choice matters, but it helps to keep the decision grounded.

In practice, you usually care about:

  • does it retrieve the right chunks on your questions;
  • does it handle your language mix well;
  • is it fast enough for your workload;
  • is the vector size reasonable for storage and latency;
  • can you run it where you need to run it.

Public benchmarks are useful, but your own corpus matters more. A model that looks strong on a leaderboard may still be awkward for your documents.

For this project, the default embedding model is bge-m3:latest. That is a practical baseline, not a claim that one model is always best.

3.11 The Difference Between an Embedding, an Index, and a Vector Database

These terms are easy to blur together, so it is worth separating them clearly.

An embedding model:

  • turns text into vectors.

An index:

  • is the searchable data structure built from those vectors.

A vector database:

  • is a storage and retrieval system built to manage vector indexes, metadata, updates, and search at larger scale.

In a small project, these can all feel like one thing because the same script may handle all of them. But they are different layers.

This repository has:

  • an embedding model;
  • a simple local JSON index;
  • no full vector database yet.

That distinction matters because it helps you see what problem each tool is solving.

3.12 Why ANN Search Exists

If a project has only a small number of chunks, it is possible to compare a question embedding against every chunk directly. That is basically what this project does now.

But as the corpus grows, that gets slower and more expensive.

Approximate nearest-neighbor search, usually shortened to ANN, is the family of methods used to make vector search faster at larger scale. The core tradeoff is simple:

  • exact search tries harder to find the very best matches;
  • approximate search is faster, but may miss some of them.

This tradeoff is normal. The point is not to be mathematically perfect. The point is to meet the application’s quality and latency needs.

3.13 FAISS and Local Vector Search

FAISS is a common tool for local vector search. It is often a good next step after a project outgrows a hand-rolled Python loop over JSON embeddings.

Why teams like it:

  • it is fast;
  • it supports common vector search workflows;
  • it is good for experiments and local systems.

Why it does not solve everything:

  • it is not a full application platform;
  • metadata handling is more limited than in some managed systems;
  • production concerns still need extra engineering.

For this project, FAISS would be a natural upgrade because it would keep the learning setup local while making indexing and search more realistic.

3.14 Vector Databases and When They Matter

Vector databases become more useful when the retrieval system needs more than just “store some vectors and search them.”

They matter more when you need things like:

  • large corpora;
  • frequent updates;
  • metadata filtering;
  • multiple users or tenants;
  • operational guarantees around persistence and scale.

For a small teaching project, a local index is often enough. For a production system, storage and serving concerns usually become much more important.

This is why it helps to separate two questions:

  • do I need better retrieval quality;
  • do I need stronger retrieval infrastructure.

Those are related, but not the same decision.

3.15 Structured Indexing Still Matters

Not everything should be pushed into embeddings.

Some kinds of information are better handled in structured form:

  • tables;
  • records with stable fields;
  • entity lists;
  • timestamps;
  • access rules;
  • graph relationships.

If a user asks for “the latest policy after January 2025” or “all products in category X,” embeddings alone are not enough. Structured fields matter.

This is why strong RAG systems often combine vector search with metadata filters, keyword search, SQL, or other retrieval methods.

3.16 Context Expansion After Retrieval

Sometimes the best-matching chunk is still too narrow to answer the question well.

For example, the chunk may contain a definition, but not the caveat that appears in the next paragraph. In that case, the system may benefit from adding neighboring chunks or the parent section.

This is usually called context expansion.

It can help because:

  • answers often need a little more surrounding context;
  • documents often contain meaning spread across nearby chunks.

It can also hurt because:

  • more context means more tokens;
  • extra context can add noise.

So this is another design choice that should be tested, not assumed.

3.17 How to Tell Whether the Index Is Working

As with chunking, a practical question is: how do we know this part is good enough?

Some useful checks are:

  • do semantically similar questions retrieve the same useful chunks;
  • do obvious keyword questions still work well;
  • are clearly relevant chunks missing from the top results;
  • does changing the embedding model noticeably improve retrieval;
  • does the system stay fast enough as the corpus grows.

These checks help keep Chapter 3 tied to system behavior rather than turning it into a list of tools.

3.18 What This Chapter Adds to the Pipeline

Chapter 2 produced retrieval-ready chunks.

Chapter 3 turns those chunks into a searchable index.

So the handoff now becomes:

documents
-> cleaned chunks with metadata
-> embeddings
-> index
-> retrieval

That sets up the next chapter naturally. Once the system has chunks and a way to search them, the next question is no longer “How do we store information?” It becomes “How do we retrieve the right information reliably?”
